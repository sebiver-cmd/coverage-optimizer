# Migration Discipline — SB‑Optima

## Overview

SB‑Optima uses **Alembic** for database schema migrations.  All migrations
are stored in `alembic/versions/` and applied in a strict linear chain.

## Creating a new migration

```bash
# Always start from a clean, up-to-date schema:
alembic upgrade head

# Generate with autogenerate (review every diff!):
alembic revision --autogenerate -m "short_description"
```

### Naming convention

Use the pattern `NNNN_short_description.py` where `NNNN` is a zero-padded
sequence number (e.g., `0006_add_analytics_table.py`).  This is a
project-specific convention — after autogenerate creates the file, rename
it to follow this scheme.

### Review checklist

Before committing an autogenerated migration, verify:

1. **No spurious diffs** — Alembic sometimes detects column order changes or
   `server_default` differences that are dialect-specific.  Remove these.
2. **No `DROP TABLE` or `DROP COLUMN`** unless explicitly planned.
3. **Dialect compatibility** — use `sa.text("(CURRENT_TIMESTAMP)")` for
   `server_default` rather than `func.now()` to keep SQLite compatibility.
4. **Foreign keys** reference existing tables (check migration order).
5. **Imports** — ensure any custom types (e.g., `_GUID`) are imported.

## Running migrations

### Development (local)

```bash
export DATABASE_URL="sqlite:///dev.db"
alembic upgrade head
```

### Staging / Production (Postgres)

```bash
export DATABASE_URL="postgresql://user:pass@host:5432/sboptima"
alembic upgrade head
```

> **Tip:** In Kubernetes / Docker Compose, run migrations as an init
> container or entrypoint script *before* starting the app.

### Rollback

If a migration needs to be reverted:

```bash
alembic downgrade -1        # one step back
alembic downgrade <rev_id>  # to a specific revision
```

> Not all migrations support downgrade.  Always verify the `downgrade()`
> function exists in the revision file before running.

## CI safety check

The test `tests/test_migrations_sqlite.py` validates migration safety as
part of the normal `pytest` suite.  It verifies:

- The ORM schema (via `Base.metadata.create_all`) produces all expected
  tables on SQLite — confirming model consistency.
- The Alembic revision chain is strictly linear (no branching).
- Every migration file is importable with valid `upgrade()`/`downgrade()`.
- Revision IDs are unique across all migration files.

The CI check is **required to pass** before merging.

## Best practices

- **One migration per PR** — keep the diff reviewable.
- **Never edit committed migrations** that have been applied in staging
  or production.  Create a new migration instead.
- **Linear chain only** — avoid merge migrations.  If branches diverge,
  coordinate to rebase before merging.
- **Test with SQLite first** — if it works on SQLite, it will almost
  certainly work on Postgres.  Edge cases (e.g., `ALTER TABLE` limitations
  on SQLite) are rare and well-known.
